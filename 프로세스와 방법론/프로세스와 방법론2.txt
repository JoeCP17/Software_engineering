좋은 프로세스의 특성 

1. 예측이 가능해야한다.

2. 테스팅과 유지보수 용이성

-> 코딩에 많은 노력을 기울이는것보다 요구 분석 , 설계와 테스팅에 많은 노력을 기울여야한다.


3. 변경지원  : 변경을 쉽게 다룰 수 있는 프로세스 


4. 결함제거가 용이하다.


<프로세스 모델 > 

(프로세스 모델) 
-일반적인 모델이 될만한 프로세스를 기술한 것을 말한다.

(대표적인 프로세스 모델)
 
-폭포수 모델 (계단식폭포형태를 생각하면됨) 
1970년대 소개가 되었다. 

각 단계가 다음 단계 시작전에 끝나야한다. 
-순서적 - 각 단계 사이에 중복이나 상호작용이 없다. 
-각 단계의 결과는 다음 단계가 시작되기 전에 점검 
-바로 전단계로 피드백 (정확히 하면 전단계밖에 피드백이 가능하다) 


폭포수 모형의 장단점 

장점 
-프로세스가 단순하여 초보자가 쉽게 적용 가능 
-중간 산출물이 명확, 관리하기 좋다.
-코드 생성 전 충분한 연구와 분석 단계 

단점 
-소용 없는 다종의 문서를 생산할 가능성이 있다.
-애매한 부분이 남아 있거나 프로세스 진행 과정에 변경될 수 있는데 이를 수용할 수 없다.
-테스트 작업이 프로젝트 후반 , 즉 시스템이 완성된 후에 시작된다.

적용 
-이미 잘 알고 있는 문제나 연구 중심 문제에 적합하다.
-변화가 적은 프로젝트에 적합하다.

<V모델 : 폭포수 모델 확장형> 
검증을 강화하는 관점에서 폭포수 모델을 확장한 모델 

* 기사시험때 각 검증을 빈칸을 두고 적으라함 

밑에서부터 
모듈 검증 -> 인터페이스 검증 -> 시스템 검증 -> 요구분석 검증 순 

폭포수 모형의 변형
-감추어진 반복과 재 작업을 드러낸다.
-작업과 결과의 검증에 초점 

장점 
-오류를 줄일 수 있다.

단점 
- 반복이 없어 변경을 다루기가 쉽지 않다.

적용
-신뢰성이 높이 요구되는 분야



<프로토타이핑 모델>

프로토 타이핑
-요구 사항에 대한 피드백을 받기 위해 시스템을 실험적으로 만들어 사용자에게 보여주고 평가하게
 하는 방법

프로토타이핑 도구
-시스템의 작동을 시뮬레이션 하여 사용자가 볼 수 있는 반응을 보여준다.

공동의 참조 모델
-사용자와 개발자의 의사소통을 도와주는 좋은 매개체 

프로토타입의 목적
-단순한 요구 추출 :  만들고 버린다.
-제작 가능성 타진 :  개발 단계에서 유지보수가 이루어진다.

장점 
-사용자의 의견 반영이 잘 된다.
-사용자가 더 관심을 가지고 참여할수 있고 개발자는 요구를 더 정확히 도출할 수있다.

단점 
- 비용이 많이 든다. 
-오해, 기대심리 유발
-관리가 어렵다.(중간 산출물 정의가 난해) 

적용
-개발 착수 시점에 요구가 불투명할때
-실험적으로 실현 가능성을 타진해 보고 싶을 때 
-혁신적인 기술을 사용해 보고 싶을 때 

ex. 아파트 , 패션쇼 등 고객들의 반응을 볼때 사용한다.

<진화적 모델>
점증적으로 개발하거나 반복적으로하여 각각의 완성도를 높이는 방법 

개발 사이클이 짧은 환경
-빠른 시간 안에 시장에 출시하여야 이윤에 직결
-개발 시간을 줄이는 법 -시스템을 나누어 릴리스 한다.

릴리스 구성 방법
-점증적 방법 : 기능별로 릴리스
-반복적 방법 : 릴리스 할 때마다 기능의 완성도를 높인다. 

릴리스를 나눠 하는 이유 
-시스템을 개발하는 도중에 추가 반영하여 수정하기 쉽다.
-리스크가 적기때문에 안정적인 운영이 가능하다.

장점 
-몇가지 기능이 부족하더라도 초기에 사용 교육 가능 
-사용자의 요구를 빠르게 반영한다.
-새로운 기능을 가진 소프트웨어에 대한 시장을 빨리 형성
-가동 중인 시스템에서 일어나는 예상하지 못했던 문제를 신속하고 꾸준하게 고쳐 나갈 수 있다.

단점
-프로젝트 관리가 복잡해지기때문에 작은 프로젝트 부적합
-끝이 안보일수 있어 실패의 위험이 커진다.
-프로젝트의 진행이 위험 분석에 크게 의존한다. 

<나선형 모델> 

- 진화적 프로토타입의 대표적인 모델 

소프트웨어의 기능을 나누어 점증적으로 개발 
-실패의 위험을 줄인다.
-테스트 용이
-피드백 

여러 번의 점증적인 릴리스(incremental releases) 

Boehm이 제안

반복 순환 단계 
1. 목표, 방법 , 제약 조건 결정
2. 위험 요소 분석 및 해결
3. 개발과 평가
4. 다음 단계의 계획 

장점 

1. 대규모 시스템 개발에 적합하다.
2. 반복적인 개발 및 테스트 -강인성 향상
3. 한 사이클에 추가 못한 기능은 다음 단계에 추가 가능하다.


단점 

1. 관리가 복잡하다.
2. 위험 분석을 잘못하여 지나친 경우 피해가 크다.
3. 성공 사례가 많이 알려지지 않았다.

적용 

1. 재정적 또는 기술적으로 위험 부담이 큰 경우 
2. 요구 사항이나 아키텍처 이해에 어려운 경우 

* 성공 사례가 그리 많진 않다.

<애자일 프로세스> 

폭포수 프로세스의 단점을 해결 

절차와 도구보다 개인과 소통을 중요시 한다.

1~4주간의 짧은 주기로 개발을 반복한다.

실행되는 소프트웨어를 개발하여 단계적으로 시스템 전체를 완성 - 테스트 중심 개발 


(애자일 선언) 

1. 프로세스나 도구 보다는 개인과 상호작용

2. 포괄적인 문서보다는 작동하는 소프트웨어

3. 계약에 대한 협상보다는 고객과의 협력

4. 계획을 고수하기 보다는 변화에 대응 

이 네 가지를 더욱 가치있게 여긴다.


(스크럼)

-개발 팀원 모두가 함께 소통하고 협력하여 짧은 주기를 반복하며 소프트웨어를 
 개발하는 작업 , 역할 , 결과물 


-백로그를 정하고 여기에 우선순위를 부여한다.


-짧은 주기 (스프린트)  : 1~4주의 반복기간을 의미한다. 


(스프린트 계획)

-스프린트 동안 수행할 스프린트 목표와 이를 이루기 위한 작업 상세 내역을 정한다.

- 작업 상세 내역은 스프린트 동안 수행할 작업을 목록화한 스프린트 백로그로 정리

-스프린트 내 완료기준을 정하고 팀원들과 공유


(백로그)

-스프린트 목표를 달성하기 위한 작업 목록 

(일일미팅) <Daily Scrum>

- Scrum Master가 주재하여 매일 15분 정도 논의

- 지난 스크럼 미팅 이후 수행한 작업 , 오늘부터 다음 스크럼 미팅까지 수행할 작업

- 문제있는 부분에 대한 토론 

-> 이러한 작업이 이루어지는 방을 War Room이라고 한다. 


(Burndown Chart) 
-매일 팀이 완료할 작업이 얼마나 남았는지에 대한 새 추정치를 보여준다. 

-이상적으로 이 차트는 스프린트의 마지막 날에 "남은 작업이 0"이 되는 방향의 감소 그래프 

(스프린트 리뷰)

-릴리스 될 Increment에 대한 발표
-스프린트의 결과를 이해하고 백로그에 대한 변경 필요성 파악

-> 프로젝트 오너가 완성된 백로그와 완성되지 않은 백로그 설명

-> 다음 스프린트에서 무엇을 할 것인지 토의

Given - When - Then 순으로 


스크럼 진행 순서 

1. Po는 제품의 요구 기능(User Stroy)과 우선 순위를 제품 백로그로 정한다.

2. PO가 정한 제품의 우선순위에서 어디까지 작업을 할지 팀과 조율

3. 스프린트 목표를 구현 가능하도록 팀에서 스프린트 백로그를 작성한뒤 작업 할당

4. 스프린트를 진행하는 동안, 매일 정해진 장소와 시간에 모든 개발 팀원이 참여하는
   일일 스크럼 회의를 가진다.

5. 매회의 스프린트가 종료할 때마다, 스프린트 리뷰(Review)를 통해 만들어진 제품을
   검토하고 개선사항을 이해 한다.

6. 제품의 리뷰를 통해 제품의 지속적 개선사항 도출이 끝나면,스프린트 회고를 통해
   팀의 개발 문화(프로세스)에 대한 개선의 시간을 갖는다.

7. 다음 스프린트에서 수행할 백로그를 PO와 필요 인원이 모여 선정하고 계획하는 
   시간을 갖는다.
