소프트웨어 프로젝트 
-수행할 작업을 조직화한 프로세스를 이용
-비용,일정,품질에 대한 목표를 성취하는 것 

프로세스 명세 
-프로젝트에서 수행하여야 하는 작업과 이들의 수행순서를 정의하는 것 
-실행 프로세스는 다를 수 있다. 

프로세스 모델 
-일반적인 프로세스를 기술하는 것을 이야기한다.
-작업의 단계와 순서를 이야기한다. 
-각 단계 작업수행의 제약사항이나 조건을 모아 놓은것 


SDLC : software development life cycle 

-소프트웨어 개발 프로세스는 소프트웨어가 탄생되어 운용되고 유지보수를 거쳐 
 폐기되기 까지 전 과정을 의미한다.

- 인간의 탄생과 죽음에 이르기까지의 과정과 유사하다. 

순서 : 요구분석 -> 설계 -> 구현 -> 테스팅 -> 유지보수

- 서로 협력하여 전체 목적을 만족

요구 분석 
ㄴ시스템이 사용자에게 제공해야하는것들이 무엇인가 응용분야에 집중해야 하는 단계 

요구 - 시스템이 가져야 할 능력과 조건 

what의 단계

결과물 : 요구분석서 = 요구분석 명세서 (SRS : Software Requirement Specification)

통합과 테스트
 
병행 : 통합해 나가면서 테스트 시작 

모듈에 통합으로 시작 

점차 완성된 모듈을 추가

통합은 개발자가 주로 담당한다.

테스트는 QA 팀이 주로 담당 

단계적인 테스트 
- 단위 , 통합 , 시스템 

목적 중심 테스트 

-스트레스 테스트  : 개발한 시스템에 오버로드를 하는 테스트 
                        하는 이유 : 최악의 상황을 견디기 위한 테스트 

-성능 테스트 : 제대로 시스템이 작동되는지 확인하는 테스트 

- 베타 테스트 : 조직외 테스트 해보는것을 말한다. (ex.게임 출시전 베타테스트 생각) 
- 알파 테스트 : 개발 조직내에서 테스트하는것을 말한다. 
 
- Acceptance 테스트 : 제품을 다 만든후 사용자에게 전달하였을때 제대로 되는지 테스트해보는 테스트

- Usability 테스트 : 시스템을 사용해서 우리가 제대로 효과를 보고있는지 확인하는 테스트
